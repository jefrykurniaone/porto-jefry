## What Was Done

### Fix 1: ThemeProvider Nonce (layout.tsx)
Added `import { headers } from 'next/headers'` and read the `x-nonce` request header inside `LocaleLayout`. Passed the nonce as a prop to `<ThemeProvider nonce={nonce}>`. The nonce is `undefined`-safe — falls back to `undefined` if header is absent, which ThemeProvider handles gracefully.

**Before:**
```tsx
<ThemeProvider attribute='class' defaultTheme='system' enableSystem>
```
**After:**
```tsx
const nonce = (await headers()).get('x-nonce') ?? undefined;
// ...
<ThemeProvider attribute='class' defaultTheme='system' enableSystem nonce={nonce}>
```

### Fix 2: `unsafe-inline` for style-src (middleware.ts)
Added `'unsafe-inline'` to the production `style-src` directive in `buildCsp()`.

**Before:**
```ts
`style-src 'self' 'nonce-${nonce}' https://fonts.googleapis.com`,
```
**After:**
```ts
`style-src 'self' 'unsafe-inline' 'nonce-${nonce}' https://fonts.googleapis.com`,
```

## Surprises

1. **next-themes v0.4.x nonce support was not wired up** — the ThemeProvider nonce prop was added in next-themes v0.4, but the layout never connected it to the middleware-generated nonce. This was a silent failure (script blocked, theme still applied via fallback).

2. **Next.js 14.2.35 does not auto-nonce `next/font` inline styles** — despite the middleware setting `x-nonce`, the `@font-face` inline `<style>` generated by `Inter()` did not receive the nonce. This is a known limitation: `Inter()` is called at module level, outside the per-request render context.

3. **Error 3 was a browser extension artifact** — the "message channel closed" error looked like a serious async error but was completely external to the website.
