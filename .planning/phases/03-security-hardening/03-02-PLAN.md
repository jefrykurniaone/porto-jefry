<objective>
Implement SEC-01 (trusted client-IP rate-limit identity) and the SEC-03 accepted-risk
documentation. Replace the spoofable two-header IP fallback chain in the CV-download route
with the official `ipAddress()` helper from `@vercel/functions` (D-01, D-02), preserving the
`'127.0.0.1'` local-dev fallback (D-03) and leaving all rate-limit logic untouched (D-04).
Add behavioral tests (SEC-01-a/b/c) and formally close SEC-03 as an accepted-risk in two
documentation locations (D-12, D-13, D-15).

Purpose: A client-supplied `x-forwarded-for` header currently can set the rate-limit bucket key,
allowing an attacker to bypass the per-IP limit by rotating that header. `ipAddress()` reads only
the Vercel-infrastructure-injected `x-real-ip`, which clients cannot forge.
Output: Updated `route.ts`, new `route.test.ts`, `@vercel/functions` in `package.json`, and a
closed SEC-03 entry in `REQUIREMENTS.md`.
</objective>

| Boundary | Description |
|----------|-------------|
| client -> /api/generate-cv | Untrusted HTTP request crosses into the rate-limited CV render endpoint. Request headers (including x-forwarded-for) are attacker-controllable; only Vercel-injected x-real-ip is trustworthy. |
| Vercel edge -> route handler | Vercel infrastructure sets x-real-ip from the real TCP client IP before the request reaches the Node handler. |

| T-3-01 | Elevation of Privilege | IP extraction in GET /api/generate-cv (route.ts:99-102) | mitigate | Replace the x-real-ip/x-forwarded-for header chain with `ipAddress(req)` from @vercel/functions, which reads only the infra-injected x-real-ip header — a client-supplied x-forwarded-for can no longer reset the rate-limit bucket (Task 2; D-01/D-02). |
| T-3-03 | Denial of Service | CV render path (sharp + @react-pdf/renderer) behind in-memory rate limiter | accept | Per-instance Map limiter retained (D-12); the expensive render is module-cached per locale (CV_BUFFER_CACHE) so it runs at most once per instance lifetime.
