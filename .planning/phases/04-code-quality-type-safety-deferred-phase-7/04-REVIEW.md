## Summary

This phase introduced build-time LAST_MODIFIED_DATE codegen (`scripts/gen-build-meta.mjs`), type-safety workaround removal in Experience (replacing `useMessages` with `t.raw`), tech-array deduplication in `src/data/projects.ts`, function-length refactors (sub-components extracted in Hero, Navbar, Contact, About, Projects, Experience), and hook extractions (`use-cv-download`, `use-focus-trap`, `use-scrolled`).

The refactors are generally sound. Two critical issues were found: (1) `gen-build-meta.mjs` silently destroys any future constants beyond `BASE_URL` by reconstructing the entire file from a single-line scan, and (2) the `t` prop threaded through `HeroCtaButtons` is typed with a hand-rolled alias rather than the actual `next-intl` return type, causing a type mismatch that TypeScript accepts but that will break at runtime or under strict call-site analysis. Five warnings cover a stale `key` prop, a focus-trap snapshot bug, the missing `link.remove()` after programmatic click, test brittleness, and the Contact component using `React.ReactNode` without importing React. Four informational items round out the report.

## Critical Issues

### CR-01: `gen-build-meta.mjs` destroys all constants except `BASE_URL` on every build

**File:** `scripts/gen-build-meta.mjs:63-87`

**Issue:** `updateConstants` reads the file, extracts only the first line that starts with `export const BASE_URL`, then **overwrites the entire file** with exactly two exports: `BASE_URL` and the newly generated `LAST_MODIFIED_DATE`. Any other constant that a future developer adds to `constants.ts` (e.g. `SITE_NAME`, `ANALYTICS_ID`, `OG_IMAGE_PATH`) is silently deleted the next time `npm run build` is run. The file also loses any leading comment blocks, type annotations, or blank-line formatting beyond the single blank line the script injects.

The current file only has two exports, so the bug is latent but will trigger the moment any third constant is added — and the symptom will be a confusing TypeScript "cannot find name" error at an unrelated call site, not at `constants.ts` itself.

**Fix:** Use a targeted regex replacement instead of a full rewrite. Replace only the `LAST_MODIFIED_DATE` line (or append it if absent), leaving all other lines untouched:

```javascript
function updateConstants(date) {
    const existing = readFileSync(CONSTANTS_PATH, 'utf8');
    const replacement = `export const LAST_MODIFIED_DATE = '${date}';`;

    const DATE_LINE_RE =
        /^export const LAST_MODIFIED_DATE\s*=\s*'[^']*';$/m;

    let updated;
    if (DATE_LINE_RE.test(existing)) {
        // Replace the existing line in-place
        updated = existing.replace(DATE_LINE_RE, replacement);
    } else {
        // Append after a generation comment (first-time run)
        updated =
            existing.trimEnd() +
            '\n\n// Generated by scripts/gen-build-meta.mjs — do not edit manually\n' +
            replacement + '\n';
    }

    writeFileSync(CONSTANTS_PATH, updated, 'utf8');
    process.stdout.write(
        `[gen-build-meta] LAST_MODIFIED_DATE set to ${date} in src/utils/constants.ts\n`
    );
}
```

### CR-02: `t` prop in `HeroCtaButtons` uses an under-constrained hand-rolled type alias

**File:** `src/components/sections/Hero.tsx:13` and `src/hooks/use-cv-download.ts:11`

**Issue:** Both files define an identical local `TranslateFn` alias:

```typescript
type TranslateFn = (key: string, values?: Record<string, string | number>) => string;
```

The `t` function returned by `useTranslations('hero')` does **not** match this signature. The next-intl `useTranslations` return type is a strongly typed function keyed to the message namespace; its `values` parameter type is `Record<string, string | number | boolean | Date | ReactElement | ...>`, not the narrow `Record<string, string | number>` declared here. Additionally, it does not return `string` in all cases — it can return `ReactElement` when the message contains React components.

The consequence: TypeScript accepts the `t={t}` prop because it widens the return type to `string` (the component body only calls `t('cv_error', { status })` with a plain string value, so no runtime crash today), but the alias creates a false impression of type safety and will produce silent failures if any hero translation key ever uses a React element interpolation. The duplicate alias definition between Hero and `use-cv-download` also violates DRY and means any fix must be applied twice.

**Fix:** Remove the local `TranslateFn` alias from both files and import the canonical type from next-intl. Pass `t` as `ReturnType<typeof useTranslations>` or use a narrower but accurate form:

```typescript
// In use-cv-download.ts
import type { useTranslations } from 'next-intl';
type TranslateFn = ReturnType<typeof useTranslations>;

export function useCvDownload(locale: string, t: TranslateFn): UseCvDownloadReturn {
```

```typescript
// In Hero.tsx — remove the local TranslateFn alias entirely
// Change HeroCtaButtonsProps:
interface HeroCtaButtonsProps {
    locale: string;
    ctaWork: string;
    ctaCv: string;
    ctaContact: string;
    ctaDownloading: string;
    t: ReturnType<typeof useTranslations>;
}
```

This removes the alias duplication and restores proper type coverage.

### WR-02: `use-cv-download.ts` appends an `<a>` to `document.body` implicitly via `link.click()` but never removes it

**File:** `src/hooks/use-cv-download.ts:25-30`

**Issue:**
```typescript
const link = document.createElement('a');
link.href = url;
link.download = 'Jefry_Kurniawan_CV.pdf';
link.click();
URL.revokeObjectURL(url);
```

The programmatic `link.click()` pattern works reliably only when the element is appended to the document. In some browsers (older Firefox, Safari) a detached anchor's `click()` does not trigger a download. More importantly, even in browsers where it works today, the element is created, clicked, and leaked into the function scope without cleanup. The correct cross-browser pattern appends to `document.body`, clicks, then removes. As written, the download may silently fail in certain environments.

**Fix:**
```typescript
const link = document.createElement('a');
link.href = url;
link.download = 'Jefry_Kurniawan_CV.pdf';
document.body.appendChild(link);
link.click();
document.body.removeChild(link);
URL.revokeObjectURL(url);
```

### WR-03: `use-focus-trap` captures focusable elements at mount time — DOM mutations after open will miss new elements

**File:** `src/hooks/use-focus-trap.ts:51-57`

**Issue:** The focusable element list is captured once when the `useEffect` runs (i.e., when `isOpen` first becomes `true`):

```typescript
const focusable = Array.from(
    menu.querySelectorAll<HTMLElement>('a, button, [tabindex]:not([tabindex="-1"])')
);
```

This snapshot is then closed over inside `handler`. If the mobile menu's DOM is modified after the effect runs (e.g., a sub-menu expands, or a SGDS web component hydrates and adds focusable shadow children), the Tab-trap will operate on a stale list. In particular, `<sgds-*>` web components render shadow DOM: native `querySelectorAll` does not pierce shadow roots, so any focusable elements inside SGDS components (buttons rendered in shadow DOM) are already missing from `focusable`. This means `Tab` may escape the trap by reaching a shadow-DOM button that is not in the snapshot.

The Escape-key path is unaffected (it calls `onClose()` unconditionally). The bug manifests only when a user Tabs to the last visible element and the shadow-DOM button was not captured.

**Fix:** For the shadow DOM issue, either add `::slotted` query or re-query on each `Tab` keypress rather than snapshotting:

```typescript
const handler = (e: KeyboardEvent) => {
    // Re-query on each keydown to handle dynamic DOM and shadow elements
    const currentFocusable = Array.from(
        menu.querySelectorAll<HTMLElement>(
            'a, button, [tabindex]:not([tabindex="-1"])'
        )
    );
    handleFocusTrapKeyDown(e, currentFocusable, onClose, toggleRef);
};
```

### WR-04: Experience source-inspection tests use `fs.readFileSync` with a relative path — fragile if `vitest` CWD changes

**File:** `src/components/sections/Experience.test.tsx:59` and `src/components/sections/Hero.test.tsx:176,189`

**Issue:** Both test files call `fs.readFileSync('src/components/sections/Experience.tsx', 'utf-8')` (and similar relative paths). These calls resolve relative to `process.cwd()`, which Vitest sets to the project root when invoked via `npm run test`. However, if tests are ever run from a different directory, or if `vitest.config` specifies a different `root`, the relative path will fail with `ENOENT`. Additionally, source-inspection tests (checking that source contains or lacks certain strings) are inherently brittle: they test the build artifact rather than behavior and will silently pass even if the file is incorrectly structured, provided the string is present.

These tests also violate the CLAUDE.md guideline "Test behavior, not implementation details."

**Fix:** Use `import.meta.url` to build an absolute path if source-inspection tests are retained:
```typescript
import { fileURLToPath } from 'url';
import { dirname, join } from 'path';

const __dirname = dirname(fileURLToPath(import.meta.url));
const source = fs.readFileSync(join(__dirname, 'Experience.tsx'), 'utf-8');
```
Or, preferably, convert these tests to behavioral assertions against the rendered output.

### WR-05: `Contact.tsx` uses `React.ReactNode` in the `ContactLinkItem` interface without importing React

**File:** `src/components/sections/Contact.tsx:18`

**Issue:**
```typescript
interface ContactLinkItem {
    ...
    icon: React.ReactNode;
    ...
}
```

`React` is referenced as a namespace but is never imported. In Next.js 14 with the new JSX transform, React does not need to be imported for JSX — but using `React.ReactNode` as a type still requires `React` to be in scope (either via `import React from 'react'` or `import type { ReactNode } from 'react'`). TypeScript with `"jsx": "react-jsx"` will not error on this because `React` is injected into the global scope by the JSX transform in the compiled output, but the type-checker may or may not resolve `React.ReactNode` depending on `tsconfig` settings (`"moduleResolution"` and whether `@types/react` globally augments the `React` namespace). This is a latent type error that currently passes only because `@types/react` populates `React` as a global namespace under certain configurations.

**Fix:** Replace with an explicit import:
```typescript
import type { ReactNode } from 'react';

interface ContactLinkItem {
    ...
    icon: ReactNode;
    ...
}
```

**File:** `scripts/gen-build-meta.mjs:81`

### IN-03: `use-scrolled.ts` initialises `isScrolled` to `false` without reading the current scroll position — mismatch on SSR-to-client hydration

**File:** `src/hooks/use-scrolled.ts:10`

**Issue:** If the page loads at a scroll position already past the threshold (e.g., user clicks a deep anchor link or the browser restores scroll position), `isScrolled` will be `false` on the first render and only corrected after the first scroll event fires. For the Navbar, this means the transparent/opaque class swap will be wrong on initial paint for users whose page loads mid-scroll. This is a minor visual glitch rather than a functional bug.

useEffect(() => {
    // Set correct initial state
    setIsScrolled(globalThis.scrollY > threshold);

    let ticking = false;
    const onScroll = () => { ... };
    ...
}, [threshold]);

### IN-04: `error.tsx` sub-component `ErrorActions` passes `locale` to construct a manual `/${locale}` href — bypasses next-intl's `Link` routing for locale prefix

**File:** `src/app/[locale]/error.tsx:25`

**Issue:**
```tsx
<Link href={`/${locale}`}>
```
The project uses next-intl's typed `Link` (imported from `@/i18n/routing`). next-intl's `Link` already knows the current locale and prepends it automatically — callers should pass `href="/"` (the root path without locale prefix) and let next-intl handle prefix injection. Manually constructing `/${locale}` means the locale is prepended twice if next-intl's routing is configured in prefix mode, producing a broken URL like `/en/en`.

Whether this is actually broken depends on the routing configuration in `src/i18n/routing.ts`. If `localePrefixMode` is `'always'` (the default), the double-prefix bug is active. If it is `'as-needed'` with `en` as the default locale, it may accidentally work for `en` but fail for `id`.

**Fix:** Pass a plain `href="/"` to next-intl's `Link` and remove the `locale` prop from `ErrorActions`:
```tsx
<Link href="/">
    <sgds-button type='button' variant='outline' suppressHydrationWarning>
        {returnHomeLabel}
    </sgds-button>
</Link>
```
Remove the `locale` parameter from `ErrorActionsProps` and the `ErrorActions` function signature.
