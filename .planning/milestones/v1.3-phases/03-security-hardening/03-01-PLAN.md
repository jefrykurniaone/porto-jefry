# Phase 3 — Plan 01: Ad-hoc CSP Hotfix (Production Console Errors)

**Status:** Complete  
**Type:** Hotfix (triggered by observed production errors, pre-dates formal Phase 3 execution)  
**Date:** 2026-06-03  
**PR:** https://github.com/jefrykurniaone/porto-jefry/pull/21

---

## Context

While reviewing the production site (porto-jefry.vercel.app/en), three console errors were observed in DevTools. Two were real CSP violations; one was a browser extension artifact.

## Problem

### Error 1 — CSP Inline Script Violation
```
Executing inline script violates Content Security Policy directive 'script-src'
'nonce-...' required — action has been blocked.
```
**Root cause:** `next-themes` `ThemeProvider` injects a blocking inline `<script>` (before React hydrates) to prevent flash of wrong theme. `LocaleLayout` never read or passed the per-request nonce from `x-nonce` request header to `ThemeProvider`, so the script was blocked.

### Error 2 — CSP Inline Style Violation
```
Applying inline style violates Content Security Policy directive 'style-src'
'nonce-...' required — action has been blocked.
```
**Root cause:** `next/font/google` (`Inter`) generates an inline `<style>` tag for `@font-face` declarations. `Inter()` is called at module level, so a dynamic per-request nonce cannot be passed to it. Next.js 14.2.35 does not automatically nonce font inline style tags in App Router.

### Error 3 — Message Channel Error (NOT our bug)
```
Uncaught (in promise): A listener indicated an asynchronous response by returning true
```
**Root cause:** Browser extension artifact (password managers, ad blockers). Not from website code. No fix needed.

---

## Plan

### Fix 1 — Pass nonce to ThemeProvider (`layout.tsx`)
- Import `headers` from `next/headers`
- Read `x-nonce` request header inside the server component
- Pass as `nonce` prop to `<ThemeProvider>` (next-themes v0.4.x supports this)

### Fix 2 — Allow `unsafe-inline` for `style-src` in production CSP (`middleware.ts`)
- Add `'unsafe-inline'` to `style-src` in `buildCsp()` production branch
- Rationale: CSS cannot execute code → far lower XSS risk than inline scripts; standard practice for sites using `next/font`. Strict nonce-based `script-src` is preserved.

## Threat Model Note

This hotfix adds `'unsafe-inline'` to `style-src`. This was a conscious trade-off:
- CSS injection cannot execute arbitrary JavaScript
- It unblocks font rendering without maintaining fragile static hashes
- `script-src` remains strict with per-request nonces and `'strict-dynamic'`
- The formal Phase 3 SEC-02 goal (remove `fonts.googleapis.com`) is still pending
