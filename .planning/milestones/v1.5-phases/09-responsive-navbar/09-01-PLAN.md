<objective>
Build the self-contained, accessible mobile navigation drawer and its supporting primitives WITHOUT touching Navbar.tsx. This plan delivers a new `MobileDrawer` component (right-side slide-in panel), a `useScrollLock` hook, and the new bilingual i18n labels the drawer needs. Plan 02 wires this drawer into the Navbar.

Purpose: Isolating the drawer + hooks + i18n into new files lets this run as a clean foundation wave with zero file overlap against the Navbar integration, keeps Navbar.tsx under the 300-line budget, and makes the drawer's accessibility behavior independently unit-testable.

Output: `MobileDrawer.tsx` + co-located test, `use-scroll-lock.ts`, and new `nav.*` keys added to BOTH `en.json` and `id.json`.

Drawer-mechanism decision (REQUIRED justification for NAV-02): The locked design names `sgds-drawer` as the preferred pattern but explicitly permits a custom React panel "if `sgds-drawer` cannot meet the a11y criteria." It cannot meet them in THIS codebase's verification path: in the Vitest/jsdom environment every `sgds-*` tag is registered as a bare `HTMLElement` stub (see `src/test/setup.ts`) with no Lit upgrade, so `sgds-drawer`'s built-in focus trap, overlay/backdrop, scroll lock, and Esc handling never execute under test — making A11Y-01..04 impossible to assert and uncoverable against the coverage gate. A React-owned panel instead reuses the already-proven `use-focus-trap.ts` hook (React owns `isOpen`, the panel ref, and the toggle ref), keeps the existing passing focus-trap/Esc test approach, and meets the same visual contract (right-side slide-in). This is the justified fallback. Do NOT introduce `sgds-drawer`.
</objective>
