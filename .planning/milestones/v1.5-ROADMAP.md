# Roadmap Archive: porto-jefry — v1.5 Responsive Navigation & Layout Fixes

**Shipped:** 2026-06-21 · **Tag:** v1.5 · **PR:** #36 (squash)
**Phases:** 2 (9 → 10) · **Plans:** 3 (09-01, 09-02, 10-01)
**Requirements:** NAV-01..05, A11Y-01..04, LAYOUT-01..02 (11 total, all complete)

## Summary

Fixed the prod-mobile regressions reported on a Samsung Galaxy S8 (360×740): the hero photo was clipped under the fixed navbar, only 3 of 7 nav links were visible (the always-expanded nav overflowed with no way to scroll), and no hamburger appeared. The milestone collapses the navbar to a hamburger + accessible drawer on phones, makes the inline tablet/desktop nav horizontally scrollable whenever items overflow, and clears the hero from the fixed header on phones and tablets with no horizontal page overflow.

## Phase 9: Responsive Navbar

**Goal**: Navigation works correctly at every viewport width — phones collapse to a hamburger drawer, tablets and desktops use an inline nav that scrolls horizontally when items don't all fit
**Depends on**: Nothing (Phase 8 shipped; this began v1.5)
**Requirements**: NAV-01, NAV-02, NAV-03, NAV-04, NAV-05, A11Y-01, A11Y-02, A11Y-03, A11Y-04
**Completed**: 2026-06-21 (2/2 plans)
**Success Criteria** (all TRUE):

  1. At 360 / 390 / 430px wide, the navbar shows only the brand and the hamburger button; no inline section links are visible, and tapping the hamburger opens a right-side drawer listing all 7 sections (About, Experience, Education, Skills, Projects, Certifications, Contact) with touch targets of at least 44px
  2. Tapping any drawer link smooth-scrolls to that section, updates the URL hash, and closes the drawer — verified on both EN and ID labels
  3. The theme toggle and EN|ID language switch are present and functional inside the open drawer at all phone widths (360–430px)
  4. At 768px and 1024px (tablet), the inline nav is visible; when the 7 labels (especially longer Indonesian ones) or a non-maximized window means they don't all fit, the nav row scrolls horizontally so every section remains reachable — no item clipped or permanently hidden
  5. At 1280px+ (desktop, full-width window), all 7 nav items are fully visible with no scroll required; the drawer and hamburger are hidden
  6. While the drawer is open: keyboard focus is trapped inside it, Tab cycles within it, Esc closes it and returns focus to the hamburger button, the backdrop tap closes it, and background page scroll is locked; all interactive controls have accessible labels and meet 44px minimum touch targets

**Plans:**

- [x] 09-01-PLAN.md — Build the accessible MobileDrawer (7 links + theme/lang toggles + close/backdrop, focus-trap + scroll-lock), the `useScrollLock` hook, and bilingual `nav.close_menu` (NAV-02, NAV-04, A11Y-01..04) — Wave 1
- [x] 09-02-PLAN.md — Integrate the drawer into Navbar: collapse phone bar to brand+hamburger, remove `expand="always"` (then restored to disable SGDS's native toggler), add `overflow-x-auto` inline-nav scroll fallback, wire `scrollTo`, update tests (NAV-01, NAV-03, NAV-05, A11Y-04) — Wave 2

**Notable:** Root cause of the original bug was `expand="always"` on `<sgds-mainnav>` forcing the desktop layout at all widths. Custom React `MobileDrawer` chosen over `sgds-drawer` (jsdom stubs blocked a11y assertions on Lit components under Vitest). UAT initially failed (empty menu + stray native toggler when `expand="always"` was removed in 09-02); fixed by restoring `expand="always"` to disable SGDS's native collapse, with CSS-driven responsive behavior (commit 04db269). Focus return to toggle on all close paths (WR-01). Shipped via PR #36.

## Phase 10: Hero & Overflow Fixes

**Goal**: The hero profile photo and all page content are fully visible and horizontally contained on phones and tablets after the navbar is finalized
**Depends on**: Phase 9 (navbar height and structure must be settled before the hero top-offset is calculated)
**Requirements**: LAYOUT-01, LAYOUT-02
**Completed**: 2026-06-21 (1/1 plan)
**Success Criteria** (all TRUE):

  1. At 360 / 390 / 430px (phone, portrait) and 768 / 1024px (tablet, portrait and landscape), the hero profile photo and heading are fully visible below the fixed navbar — no part of the photo or content obscured by the fixed header
  2. On tablets in landscape orientation (e.g. 1024px wide, short viewport height), the hero content clears the fixed navbar so the photo is not clipped at the top
  3. From 320px through 1024px, no section or component causes a horizontal scrollbar; all content fits the viewport width at the validation widths 360 / 390 / 430 (phone) and 768 / 1024 (tablet), with desktop (1280px+) unaffected

**Plans:**

- [x] 10-01-PLAN.md — Fix hero top-clipping below the fixed navbar (`--navbar-height` token + `.hero-section` 100svh/top-anchored clearance with `box-sizing: border-box`) and eliminate horizontal page overflow 320→1024px (root-cause `min-width: 11.5rem` CTA fix to `max-width: 1024px` + defensive `overflow-x: clip`) (LAYOUT-01, LAYOUT-02) — Wave 1

**Notable:** Single `--navbar-height: 5rem` token is the source of truth shared by hero clearance and `section { scroll-margin-top }`. Code review added `box-sizing: border-box` (WR-01) so padding stays inside the 100svh box (project excludes Tailwind preflight). Security verification `threats_open: 0` (CSS-only). Verifier: 6/6 must-haves; human UAT 5/5 across phone/tablet/desktop in EN + ID.
